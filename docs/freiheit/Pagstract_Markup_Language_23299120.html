1. Was ist Pagstract?

Pagstract ist eine einfache Templating-Sprache, die von freiheit.com entwickelt wurde. Dabei wurde bewusst keine vollständige Programmiersprache geschaffen, sondern sich auf einfach Konstrukte, die bei der Anzeige notwendig sind, beschränkt. Dadurch bleibt die Geschäftslogik von der reinen Anzeigelogik getrennt.
In einem typischen Anwednungsfall, stellt die Anwendung die notwendigen Daten als PageModel bereit. Diese werden an ein Pagstract-Template übergeben, dass diese Daten dann in einem schönen Layout einbettet und aufbereitet.

2. Die Pagstract-Sprache

In diesem Kapitel werden Schritt für Schritt die wichtigsten und auch weniger häufig verwendete Pagstract-Konstrukte erläutert.

2.1. Hello World

Die einfachste Pagstract-Datei enthält einfach gar keine Pagstract-Befehle, sondern nur den HTML-Markup. Die generierte Seite unterscheidet sich nicht vom Template.

<html>
<head>
<title>Hello World!</title>
</head>
<body>
<h1>Hello World!</h1>
</body>
</html>

2.2. Kommentare

Kommentare können komplizierte Konstrukte verständlich machen. Natürlich sollen sie aber das generierte Markup nicht unnötig aufblähen. Dafür gibt es in Pagstract <!--- Kommentare.
Kommentare
<!-- Dies ist ein Kommentar der bleibt -->
<!--- Dies ist ein Kommentar der nur im Template sichtbar ist. -->
wird zu
<!-- Dies ist ein Kommentar der bleibt -->

Der mit <!--- begonnene Kommentar taucht also in der Ausgabe einfach nicht mehr auf.

2.3. Zugriff auf PageModel-Properties

Das PageModel ist die Basis für das Templating. Es ist die Verbindung zwischen dem Anwendungscode und dem HTML-Code. In der Anwendung wird dieses PageModel mit Werten gefüllt, auf die mithilfe von Property-Pfaden zugegriffen werden kann. Die Anwendung entscheidet, welches Template mit welchem PageModel aufgerufen wird. Das Template entscheidet, wie oder ob es diese Werte aufbereitet.

Hinweis für Anwendungsentwickler
Die TemplateFactory versucht normalerweise das Template über den Klassennamen des PageModel s zu finden. So wird das Template für de.freiheit.mein.java.pkg.TestPage PageModel unter TestPage.html gesucht. Im Umkehrschluss kann man so erraten, wie die PageModel-Klasse für ein gegebenes Template heißt.

Die Properties des PageModels können elementare Werte enthalten wie Zeichenketten und Zahlen. Sie können aber auch wiederum komplexere Unterdatenstrukturen enthalten, wie Listen oder auch wiederum Objekte, die beliebige Unterdatenstrukturen enthalten können.

Um an die Werte der Properties zu kommen, wird mit Hilfe der Punktnotation auf die Werte zugegriffen.

Beispiel: 

 .meineProperty
 .meineProperty.weitererWert

Vergleich zu Java-Beans-Pfaden

Der Syntax ist ähnlich zu den Bean-Pfaden der Java-Beans-Spezifikation ohne Index-Property-Syntax. Eine Besonderheit ist der führende Punkt der besagt, dass der Bean-Pfad nicht absolut zum PageModel, sondern relativ zum aktuellen Namespace (also möglicherweise einer Untendatenstruktur) zu sehen ist. Konstrukte, die den Namespace beeinflussen, werden später erläutert.

2.4. Ausgabe auf PageModel-Properties

Welche Daten konkret verfügbar sind, hängt natürlich von der konkreten Seite ab.

Als konkretes Beispiel, in dem auf Werte im PageModel zugegriffen werden, sieht das dann so aus:

<h1>Hello ${.name} </h1>
<h1>Hello <pma:value pma:name=".name"/> </h1>
<h1>Hello <pma:value pma:name='.name'/> </h1>
<input name="name" value="${.name}"/>
 
<!-- Ausgabe mit .name = "Peter" -->
<h1>Hello Peter</h1>
<h1>Hello Peter </h1>
<h1>Hello Peter </h1>
<input name="name" value="Peter"/>

Selbstverständlich kann hier auch auf tiefer geschachtelte Werte anhand der oben erwähnten Punktnotation zugegriffen werden.

Die "${…}"-Notation kann hierbei als Abkürzung für die explizitere pma:value-Notation (mit pma:type="plain") angesehen werden. In beiden Fällen werden normalerweise Zeichen, die eine besondere Bedeutung für HTML haben nicht einfach in den Text kopiert, sondern maskiert. Aus "<" wird somit z.B. "&lt;". Im Web-Browser erscheint somit wieder das ursprüngliche "<".

Um zu sehen, dass alle gefährlichen Zeichen maskiert werden, verwenden wir in den folgenden Beispielen einen etwas komplizierteren ".markup"-Wert. Dieser wird auch bei den anderen Maskierungsarten immer wieder als Referenz herhalten: 

<script type="text/javascript">alert('hallo');</script> T&uml;r

Wird dieser Wert nun in folgendem Template eingebunden 

<p><pma:value pma:name=".markup"/></p>

<input type="text" value="${.markup}"/>

so werden die Zeichen so umgewandelt, dass sie problemlos sowohl als Text innerhalb eines Tags als auch als Attributwert eingebunden werden können: 

<p>&lt;script type=&quot;text/javascript&quot;&gt;alert('hallo');&lt;/script&gt; T&amp;uml;r</p>

<input type="text" value="&lt;script type=&quot;text/javascript&quot;&gt;alert('hallo');&lt;/script&gt; T&amp;uml;r"/>

Der Benutzer sieht im Browser den ursprünglichen Wert, da der Browser die Umwandlung umkehrt.

2.5. Ausgabe von HTML in Properties

Der Typ pma:type="raw" und die alte Schreibweise type="raw" werden nicht mehr unterstützt. In beiden Fällen greift Pagstract auf den Typ pma:type="escaped" zurück.

Wenn eine Property HTML enthält, welches auch als solches ausgegeben werden soll, muss dieses über das pma:raw-Attribut oder als alternative Schreibweise über das pma:type-Attribut angegeben werden.

Veraltet
<pma:value pma:name=".markup" pma:type="raw"/>
oder die alte Schreibweise
<pma:value pma:name=".markup" raw="true"/>
 
<!-- Ausgabe -->
<script type="text/javascript">alert('hallo');</script> T&uml;r
oder die alte Schreibweise
<script type="text/javascript">alert('hallo');</script> T&uml;r

2.5.1 Neue HTML-Ausgabe

Um weiterhin HTML ausgeben zu können, wurde die neue Ausgabe pma:type="trustedhtml" eingeführt.

Attribute von HTML-Tags werden nichts escaped!

Text: Dies ist ein Text, der übrigens auch noch <a href="http://example.com" title="Übernachtung in Blåvand">einen Link beinhaltet & zu allem Überfluss</a> noch Sonderzeichen; weil 3 < 5
 
Ausgabe mit: <pma:value pma:name=".text" pma:type="trustedhtml"/>
 
Erzeugte Ausgabe:
Dies ist ein Text, der &#252;brigens auch noch <a href="http://example.com" title="Übernachtung in Blåvand">einen Link beinhaltet &amp; zu allem &#220;berfluss</a> noch Sonderzeichen; weil 3 &lt; 5

Warnung

Das sollte niemals für Properties geschehen, die ungefilterte Benutzereingaben enthalten. Dadurch können Sicherheitslücken entstehen!

2.6. Einfache Anfuehrungsstriche maskieren

Für das Maskieren von Sonderzeichen gibt es noch weitere Optionen.

So kann auch das Maskieren von einfachen Anführungsstrichen erzwungen werden, z.B. um eine Property innerhalb von JavaScript-Zeichenketten zu expandieren.

<!-- .stringWithSingleQuotes = "'hallo'" -->
<!-- Eingabe mit einfachen Anführungszeichen -->
<pma:value pma:name=".stringWithSingleQuotes" pma:type="escapedquotes"/>
 
<!-- Ausgabe -->
\'hallo\'

Selbstverständlich werden die anderen Zeichen mit Sonderbedeutung weiterhin so umgesetzt, dass der Benutzer den Originalstring zu sehen bekommt.

<!-- .markup = <script type="text/javascript">alert('hallo');</script> T&uml;r -->
<!-- escaped quotes -->
<pma:value pma:name=".markup" pma:type="escapedquotes"/>
 
<!-- Ausgabe -->
&lt;script type=&quot;text/javascript&quot;&gt;alert(\'hallo\');&lt;/script&gt; T&amp;uml;r
 
<!-- ALTE Variante -->
<pma:value pma:name=".markup" pma:escapeQuotes="true"/>

2.7. HTML-Tags aus Property-Werten entfernen

Ebenso gibt es die Alternative, HTML-Tags zu entfernen, anstatt sie einfach zu maskieren, so dass Markup-Code für den Benutzer direkt sichtbar wird. Das kann besonders für Benutzereingaben sinnvoll sein.

<!-- .markup = <script type="text/javascript">alert('hallo');</script> T&uml;r -->
<!-- Plain -->
<pma:value pma:name=".markup" pma:type="plain"/>
 
<!-- Plain HTML -->
<pma:value pma:name=".markup" pma:type="plainhtml"/>
 
<!-- Ausgabe -->
<!-- Plain -->
alert('hallo'); T¨r
 
<!-- Plain HTML -->
alert('hallo'); T&#168;r

2.8. Textanrisse

Häufig ist für den gesamten Text nicht immer genug Platz. Stattdessen möchte man aber einen Textanriss präsentieren. Die Maximallänge kann mit pma:maxlength angegeben werden. Es werden drei verschiedene Logiken bei Trennung unterstützt, die über pma:abbrevMode ausgewählt werden können. Wird kein oder ein nicht definierter Abkürzungsmodus gewählt, wird after als Voreinstellung verwendet.

  • strict für eine harte Trennung nach genau n Zeichen

  • after für eine weiche Trennung nach dem Wort nach mindestens n Zeichen

  • before für eine weiche Trennung vor dem Wort nach maximal n Zeichen

Über pma:more kann dem Benutzer ein Hinweis auf den weggekürzten Text gegeben werden, z.B.

<pma:value pma:name=".loremIpsum" pma:maxlength="20" pma:more="..."/>

ergibt 

Lorem ipsum dolor sit...

2.8.1. Zeichenkette für die Beispiele der unterschiedlichen Trennverhalten

.loremIpsum = "Lorem ipsum dolor sit amet"

2.8.2. Harte Trennung (strict)

<pma:value pma:name=".loremIpsum" pma:maxlength="9" pma:abbrevMode="strict" pma:more="..."/>

ergibt 

Lorem ips...

2.8.3. Weiche Trennung hinter dem Wort (after)

<pma:value pma:name=".loremIpsum" pma:maxlength="9" pma:abbrevMode="after" pma:more="..."/>
<!-- bzw. default -->
<pma:value pma:name=".loremIpsum" pma:maxlength="9" pma:more="..."/>

ergibt 

Lorem ipsum...

2.8.4. Weiche Trennung vor dem Wort (before)

<pma:value pma:name=".loremIpsum" pma:maxlength="9" pma:abbrevMode="before" pma:more="..."/>

ergibt 

Lorem...

2.9. Pagstract & XML

Pagstract orientiert sich größtenteils an XML in seinem Syntax. Die meisten Pagstract-Befehle sind dem "pma:"-Prefix bei Elementen und Attributen zu erkennen. Anders als bei XML sind aber hierfür keine expliziten Namespace-Deklarationen notwendig. Einmal geöffnete Pagstract-Befehle müssen auch wieder geschlossen werden.

Pagstract selber versucht aber nur seine eigenen Tags zu interpretieren. Der Text zwischen den Anweisungen oder in den Anweisungen wird ansonsten nicht ausgewertet. Dadurch eignet es sich auch für das Generieren nicht-XML textbasierten Formaten.

Wenngleich das zu verwirrenden Templates führen kann, müssen sich die Pagstract-Befehle aber nicht um saubere XHTML-Fragmente klammern. Z.B. (die Befehle werden später erklärt):

<p>
Lorem ipsum. Lorem ipsum.
<pma:if-visible pma:name="unterbrechMich" pma:condition="true">
</p><hr/><p>
</pma:if-visible>
Lorem ipsum. Lorem ipsum. Lorem
</p>

Wenn man dies mit unterbrechMich = true rendert, kommt folgendes heraus:

<p>
Lorem ipsum. Lorem ipsum.
 
</p><hr/><p>
 
Lorem ipsum. Lorem ipsum. Lorem
</p>

Meistens ist es aber sinnvoll, Pagstract-Befehle um symetrische XHTML-Fragmente, deren Top-Level-Elemente alle wieder geschlossen wurden, zu klammern, da man so einfacher sicher stellen kann, dass der generierte Markup fehlerfrei ist.

2.10. PageModel inspizieren

Zum Entwickeln der Template ist es hilfreich, sich das gesamte PageModel oder auch Teile davon anzuzeigen. Dazu gibt es besondere Debug-Ausgaben, mit denen man ganze PageModel-Unterbäume ausgeben kann. Dies geht aber nur auf dem Testsystem - auf dem Produktivsystem bleibt die Ausgabe leer.

Das gesamte PageModel kann man durch folgende Anweisung ausgeben und inspizieren:

<pma:debug pma:name="."/>

Einen bestimmten Unterbaum hingegen mit:

<pma:debug pma:name=".articles"/>

Die Ausgabe soll einem einen Überblick über die Struktur verschaffen. Das heißt, dass von Listen z.B. nur das erste Element ausgegeben wird.

Technischer Hintergrund: Debug-Modus einschalten
Die Java-System-Property pagstract.debug muss auf true gesetzt sein, damit pma:debug eine Ausgabe produziert.
Das passiert gewöhnlich beim Starten der Anwendung in einer Testumgebung.

2.10.1. Funktion "Wo bin ich?"

<pma:position/> gibt an, an welcher Stelle man sich aktuell im PageModel befindet, hilfreich um Verschachtelungen in Listen und Beans zu dekonstruieren.

2.11. Bean-Kontext/Namespace setzen

Um die Ausgabe einzelner Beans zu vereinfachen, kann man den Bean-Kontext oder aktuellen Namespace für alle verschachtelten Befehle auf einen Unterbaum setzen.

In folgendem Beispiel
${.teaserArticle.name}
<pma:bean pma:name="teaserArticle">
${.name}
</pma:bean>

wird zweimal der selbe Wert ausgegeben: 

Super Teaser Artikel

Super Teaser Artikel

Man kann aber auch innerhalb eines anderen Bean-Kontextes über globale Bean-Pfade auf Werte außerhalb des Bean-Kontextes zugreifen. Diese Pfade verzichten einfach auf den initialen Punkt, zum Beispiel "name". Das ist im Zweifelsfall schwer nachzuvollziehen und sollte vermieden werden.

Innerhalb einer Bean können auch Elemente außerhalb der Bean adressiert werden. Entweder durch Weglassen des führenden Punktes (Beispiel A) oder durch Abfragen eines höher liegenden Elemtens durch "../" vor der gewünschten Variable (Beispiel B)
<!-- Beispiel A -->
<pma:bean pma:name="teaserArticle">
${topLinks.home}
</pma:bean>
 
 
<!-- Beispiel B-->
<pma:bean pma:name="teaserArticle">
${../.inhalt1}
</pma:bean>

2.12 Listen

2.12.1 Einfache Listen

Der selbe Markup kann wiederholt auf jedes Element einer Liste angewendet werden. Als Basiselement wird die Property als pma:name angegeben, die die Listelemente beinhaltet. Innerhalb von pma:list beziehen sich dann alle relativen Property-Pfade auf Unterelemente des Listenelements.

<table>
<tr><th>Name</th><th>Preis</th></tr>
<pma:list pma:name=".articles">
<tr><td>${.name}</td><td>${.price}</td></tr>
</pma:list>
</table>

wird umgewandelt zu

<table>
<tr><th>Name</th><th>Preis</th></tr>
 
<tr><td>Buch 1</td><td>EUR 10</td></tr>
 
<tr><td>Buch 2</td><td>EUR 11</td></tr>
 
<tr><td>Buch 3</td><td>EUR 13</td></tr>
 
<tr><td>Buch 4</td><td>EUR 12</td></tr>
 
</table>

Leider führt das bei dieser einfachen Technik zu einem unschönen Verhalten bei leeren Listen:

<table>
<tr><th>Name</th><th>Preis</th></tr>
 
</table>

2.12.1.1 Sonderbehandlung leerer Listen

Optional kann angegeben werden, dass im Falle einer leeren Liste eine ganz andere Ausgabe erfolgt. Z.B. ein Tipp, wie die Suche generalisiert werden kann. Dazu muss der bisherige Markup in pma:header und pma:footer für Anfang und Ende; pma:content für den Markup jedes Einzelelementes und pma:no-content für den Fall einer leeren Liste angegeben werden.
<pma:list pma:name=".articles">
<pma:header><table><tr><th>Name</th><th>Preis</th></tr></pma:header>
<pma:content>
<tr><td>${.name}</td><td>${.price}</td></tr>
</pma:content>
<pma:footer></table></pma:footer>
<pma:no-content>Leider haben wir nichts passendes gefunden.</pma:no-content>
</pma:list>

wird somit mit Artikeln zu

<table><tr><th>Name</th><th>Preis</th></tr>
<tr><td>Buch 1</td><td>EUR 10</td></tr>
 
<tr><td>Buch 2</td><td>EUR 11</td></tr>
 
<tr><td>Buch 3</td><td>EUR 13</td></tr>
 
<tr><td>Buch 4</td><td>EUR 12</td></tr>
</table>

oder ohne Artikel zu 

Leider haben wir nichts passendes gefunden.

Lässt man die pma:no-content-Klausel in diesem Fall weg, so wird einfach gar nichts gerendert.

2.12.1.2 Unterscheidung gerader/ungrader Zeilen in Listen

Manchmal ist es wünschenswert im Markup zwischen geraden und ungeraden Reihen zu unterscheiden.

</pma:even>
<pma:odd>
<tr class="odd"><td>${.name}</td><td>${.price}</td></tr>
</pma:odd>
<pma:footer></table></pma:footer>
<pma:no-content>Leider haben wir nichts passendes gefunden.</pma:no-content>
</pma:list>

wird gerendert als

</table>

2.12.1.4 Markup zwischen den Zeilen einer Liste

Manchmal möchte man zwischen den einzelnen Elementen einer Liste Markup einfügen. Das geht mit pma:separator.

<pma:list pma:name=".articles">
<pma:content>
<p>${.name} für ${.price}</p>
</pma:content>
<pma:separator><hr/></pma:separator>
</pma:list>

ergibt

<p>Buch 1 für EUR 10</p>
<hr/>
<p>Buch 2 für EUR 11</p>
<hr/>
<p>Buch 3 für EUR 13</p>
<hr/>
<p>Buch 4 für EUR 12</p>

2.12.3 Virtuelle Properties für Listenelemente

Während der Iteration über die Elemente einer Liste stehen verschiedene zusätzliche bzw. virtuelle Properties bereit, die aus dem Listenkontext pro Element berechnet werden. Diese sind über den gleichen Namespace erreichbar wie die Properties des jeweiligen Listenwertes, sind von der Software aber nachträglich generiert worden und sind somit nicht wirklich im Listenelement selber enthalten. Virtuelle Properties wurden mit PUE-13323 - JIRA project doesn't exist or you don't have permission to view it.implementiert.

Property _count

Die Property _count enthält die Nummer des Eintrags in der Liste oder im gewählten Ausschnitt, immer beginnend bei 1. Diese Property ist somit relativ zum gewählten Ausschnitt, d.h. das erste Element, dass im gewählten Ausschnitt enthalten ist, erhält den _count 1, auch wenn die Gesamliste evtl. noch Elemente davor enthielt.

Property _index

Diese Property enthält den Index des Eintrags in der Gesamtliste, beginnend bei 0. Diese Property bezieht sich ungeachtet evtl. Ausschnitte immer auf die gesamte Liste, d.h. jedes Element hat unabhängig vom gewählten Ausschnitt immer den gleichen Index.

Property _total

Die Property _total enthält die Anzahl der Listeneinträge.

Property _isLast

Signalisiert das letzte Element einer Liste, bezogen auf den jeweiligen Ausschnitt. Die Property enthält den Wert true für das letzte Element und ist für alle anderen Elemente nicht verfügbar. Damit ist sichergestellt, dass sie durch ein einfaches pma:if-visible abgeprüft werden kann.

Anmerkung

2.13.1 Verkettung von Bedingungen

Seit PUE-26067 können mit einem if-visible-Tag mehrere Bedingungen gleichzeitig geprüft werden. Die Verkettung kann über die Operatoren & und | erfolgen, die allerdings nicht gemischt werden dürfen. In ihrer Funktionsweise entsprechen sie den Java-Operatoren && und ||.

2.14. Fallunterscheidung anhand von Werten

Anstatt einfach nur zu überprüfen, ob ein Wert gesetzt ist oder nicht, kann man eine Property auch gegen mehrere erwartete Werte testen.

<pma:switch pma:name=".name">

<object pma:case="Tina">
Dich kenne ich doch.
</object>
<object pma:case="Peter">
Komm Du mir nach Hause!
</object>
</pma:switch>
ergibt mit dem Wert "Peter"
    Komm Du mir nach Hause!
Wenn kein Wert für .name gesetzt ist, dann wird der Default-Wert verwendet:
    Wer bist Du denn?

2.15. Links

Links sollten entweder durch einen Resourcen-String (siehe Abschnitt 2.18, „Resourcenverweise" ) oder aus dem PageModel befühlt werden. Wenn letzteres Möglich ist, dann ist immer der Link über das PageModel vorzuziehen. Inbesondere absolute Link-Ziele führen ansonsten dazu, dass das Template nur z.B. in der Testumgebung funktionieren, dann aber im Produktivbetrieb zurück auf die Testumgebung verweisen.

Nur wenn ein Link über das PageModel befüllt wird, kann die Anwendung damit auch die richtigen Parameter mitgeben oder den Link deaktivieren.

Dazu bekommt der einfache a-Tag einen pma:name zugeordnet. 

<a href="#" pma:name=".homeLink">Startseite</a>

wird zum Beispiel zu 

<a href="http://buch.de">Startseite</a>

Für einen deaktivierten Link wird im HTML-Template anstelle eines <a>-Tag ein <span>-Tag erzeugt.

Andere HTML-Attribute wie title können ergänzt werden und bleiben in der Ausgabe erhalten: 

<a pma:name=".myAction" title="msg://foo/">Link</a>

Hinweis für Anwendungsentwickler

Das PageModel muss in diesem Beispiel eine Methode setHomeLink() besitzen, die eine Instanz von org.pagstract.model.ActionModel setzt. Diese enthält die Informationen zu Target-URL und Parametern. Außerdem liefert das ActionModel die Information, ob der Link aktiviert ist (ActionModel.isEnabled()).

2.16. Links mit Anchor

Über das Attribut pma:anchor kann ein bestimmtes Fragment direkt im Template angegeben werden, der an die im PageModel angegebene URL angehängt wird. 

<a pma:name=".homeLink" pma:anchor="targetFragment">Link</a>

ergibt somit 

<a href="http://buch.de#targetFragment">Link</a>

Hinweis

Der Inhalt des pma:anchor-Attributs wird einfach zusammen mit einem # an die im PageModel angegebene URL angehängt, d.h. es können ungültige URLs erzeugt werden, wenn die im PageModel enthaltene URL bereits ein Fragment enthält.

2.17. Image Maps

In Image Maps kann der gleiche Mechanismus verwendet werden. Hier ist die Nutzung des pma:anchor-Attributs nicht möglich: 

<map name="Landkarte">
    <area shape="rect" coord="11,10,59,29" pma:name=".myAction" />
</map>

2.18. Resourcenverweise

Es sollten keine statischen Pfade zu CSS-Dateien, Skripten oder Bildern in die Templates eingepflegt werden. Sollte sich die Basis-URL der Applikation ändern, kommt es zu Problemen. Ebenso kann das dann dazu führen, dass ein Template nur produktiv oder nur auf dem Testsystem funktioniert.

Außerdem werden Resourcen mandantenspezifisch aufgelöst und können somit auch für jeden Mandanten überschrieben werden (siehe Abschnitt 3.2, „Templates" ). 

<link rel="stylesheet" href="resource://foo/default.css" />
<link rel="stylesheet" href="resource://not/found" />

Die erste Resource konnte aufgelöst werden. Die zweite nicht, was man an dem String "null" im Attribut erkennt. Je nach Konfiguration werden aber auch nicht existieren Resourcen einfach mit einem fest konfigurierten Prefix verknüpft. 

<link rel="stylesheet" href="http://mein.server.com/basisPfad/foo/default.css" />
<link rel="stylesheet" href="null" />

Für dynamische Links aus dem PageModel muss das pma:name-Attribut verwendet werden, siehe Abschnitt 2.19, „Links" .

2.19. Formulare

Formulare, die durch die Anwendung selbst verarbeitet werden, dürfen nicht einfach durch "<form>"-Text manuell in den Markup eingefügt werden, sondern es muss der pma:form-Befehl verwendet werden. Dadurch wird sicher gestellt, dass die Zielseite richtig befüllt wird. 

<pma:form pma:name="searchForm.searchAction">
  ...
</pma:form>

wird ausgegeben als 

<form method="POST" action="http://buch.de/search" >
  ...
</form>

Hinweis für Anwendungsentwickler
Die Unterbean für das pma:name-Attribut muss vom Typ ActionModel sein. Wenn die konkrete Implementierung auch das Interface AttributeSet implementiert, dann werden die entsprechenden Attribute an das Form-Tag gehängt.

2.20. Formular input-Elemente

Das pma:name-Attribut kann auch in input-Elemten verwendet werden. Dann wird das value-Attribut richtig gesetzt und die Anwendung kann Attribute hinzufügen bzw. das Feld verstecken oder als nur lesen markieren. 

<input type="text" pma:name="name"/>
<input type="checkbox" pma:name="name" value="Peter"/>
<input type="checkbox" pma:name="name" value="Jemand anders"/>
<input name="name"  type="text"  value="Peter" >
<input name="name"  type="checkbox"  checked="checked" value="Peter" >
<input name="name"  type="checkbox" value="Jemand anders" >

Hinweis für Anwendungsentwickler

Wenn die konkrete Implementierung des Beanfeldes das Interface AttributeSet implementiert, dann werden die entsprechenden Attribute an das Form Tag gehängt.

Durch die Implementierung von ComponentModel kann gesteuert werden, wann das Element als readonly/disabled oder sogar in ein Feld des Typs hidden umgewandelt wird.

2.21. Formular select-Elemente

Pagstract hat besondere Unterstützung für Auswahlfelder mit select-Tags. 

<select pma:name="searchForm.searchType"/>

resultiert mit dem entsprechenden Model dann in 

<select name="searchForm.searchType" >
<option value="all" >Alles</option>
<option value="books"  selected>Buecher</option>
<option value="shoes" >Schuhe</option>
</select>

Hinweis für Anwendungsentwickler

Der entsprechende Wert muss das SelectionListModel unterstützen, um die Liste an Schlüssel/Wert-Paaren liefern zu können.

Der momentan gesetzte Wert kann über die getValue-Methode des SingleValueModel s festgelegt werden.

Die AttributeSet und ComponentModel Interfaces können genau wie bei input-Elementen eingesetzt werden.

2.22.1 Dynamische Variablen für Tiles definieren

Mit PUE-15640 wurde die Möglichkeit geschaffen, Variablen in einer Tile-Deklaration unterzubringen, die an die eingebundene Template weitergereicht werden. Dadurch wird es z.B. möglich, das gleiche Snippet einzusetzen, auch wenn dort z.B. zwei verschiedene CSS-Klassen benutzt werden müssten.pma:tile bietet die Möglichkeit, in einem inneren Tag pma:tileVariable per JSON zusätzliche Variablen zu definieren, die innerhalb der inkludierten Template verfügbar gemacht werden. Hierbei gilt die Einschränkung, dass in dem JSON nur einfache Werte definiert werden dürfen, enthalten Objekte und/oder Arrays werden ignoriert.

2.22.1.1 Generelle Anwendung

Die so definierten Werte werden unter dem definierten Namen, dem ein die Zeichenkette _tv vorangestellt wird, in der inkludierten Seite zur Verfügung gestellt. Dieses Präfix stellt sicher, dass es in der Unterseite nicht zu einer Überlagerung von bereits existierenden Variablennamen kommt und es wird ein Erkennungseffekt erzielt. Wird der Name bereits mit dem Präfix deklariert, wird kein zusätzliches Präfix vorangestellt.

MainPage.html
 <pma:tile filename="subpage.html">
<pma:tileVariable>
{
"prodID" : "abc",
"_tvClass" : "oCart"
}
</pma:tileVariable>
</pma:tile>
subpage.html
<a href="#" pma:name=".inWarenkorbAction" data-test="prodID-${_tvprodID}" class="${_tvClass}"></a>
Ausgabe
<a href="http://example.com/addToCart?id=1234" data-test="prodID-abc" class="oCart"></a>

2.22.1.2 Weiterreichen von Variablen

Über eine alternative Syntax im JSON lassen sich nicht nur Literalwerte sondern auch im aktuellen Namenskontext verfügbare Variablen an die Sub-Template weiterreichen. Um zwischen Literalwerten und Referenzen zu unterscheiden, wird eine leichte Abwandlung der ${...}-Syntax verwendet, die statt dem Dollar-Zeichen die Raute verwendet. Dies hat technische Gründe, der Parser akzeptiert derartige Strings mit Dollar nicht ohne umfangreichen Anpassung. Dabei ist zu beachten, dass bei der Ausgabe die normale Syntax mit Dollar verwendet werden muss.

MainPage.html
<pma:tile filename="subpage.html">
<pma:tileVariable>
{
"prodID" : "#{.articleId}"
}
</pma:tileVariable>
</pma:tile>
subpage.html
<a href="#" pma:name=".inWarenkorbAction" data-test="prodID-${_tvprodID}"></a>
Ausgabe
<a href="http://example.com/addToCart?id=1234" data-test="prodID-23602707"></a>

Dynamische Variablen können auch über mehrere pma:tile-Hierarchien weitergereicht werden. Es ist möglich, analog zum vorherigen Beispiel folgendes zu konstruieren:

subpage.html
<pma:tile filename="anotherSubpage.html">
<pma:tileVariable>
{
"prodID" : "#{_tvprodID}"
}
</pma:tileVariable>
</pma:tile>

WICHTIG: Das Weiterreichen von Variablen ist derzeit auf einfache Variablen eingeschränkt. Es können keine kompletten Objekte oder Listen bzw. Arrays weitergereicht werden. Der Renderer löst Variablenreferenzen innerhalb von pma:tileVariable immer in Strings auf, alles, was nicht oder nicht sinnvoll in einen String gewandelt werden kann, kann nicht weitergereicht werden und führt zu Fehlern.

2.22.1.3 JSON-Syntax

Da die Interpretation des Variablenblocks durch einen JSON-Parser erfolgt, muss der Inhalt auch der JSON-Syntax entsprechen. Diese kann auf http://www.json.org/ nachgelesen werden. An dieser Stelle soll nur darauf hingewiesen werden, dass es notwendig ist, String-Literale und auch Variablenreferenzen in Anführungszeichen zu kleiden. {{ "prodID" : "#{.articleId}" }} ist gültig, {{ "prodID" : #{.articleId} }} nicht. Ist das JSON nicht parsbar, wird eine Warnung geloggt und keine der enthaltenen Variablen bereitgestellt. Das eigentliche Rendering findet trotzdem statt, nur fehlen halt die deklarierten Variablen.

2.22.1.4 JSON-Encoding

Um zum jetzigen Zeitpunkt die Auswertung nicht zu verkomplizieren, liest der Parser das JSON als ISO-8859-1 ein. Das widerspricht zwar dem Sinn von JSON, derzeit sind aber alle Templates ISO-8859-1 kodiert und auch die NUTS-Templates werden zukünftig vor dem Deployment von UTF-8 zu ISO-8859-1 gewandelt, bis der Shop komplett UTF-8 fähig ist. Soll explizit ein anderes Encoding erzwungen werden, kann über das Attribut encoding an pma:tileVariable ein Encoding spezifiziert werden. Die JVM muss dieses Encoding unterstützen.

2.23. Wiederverwendbare Komponenten

Besonders praktisch für die Erstellung von wiederverwendbaren Bausteinen ist die Kombination von pma:tile mit pma:bean. Nehmen wir an wir hätten ein einfaches Template, um einen Artikel anzuzeigen:

2.24. Internationalisierung

Pagstract bietet einfache Integration von mandantenspezifischen/internationalisiertem Text in die Seite über Messages. Beispiel: 

<p>msg://foo.my.message</p>
msg://i.do.not.exist

Die erste Nachricht wird zu "Nachricht foo.my.message" aufgelöst, die zweite gibt es nicht: 

<p>Nachricht foo.my.message</p>

Wenn einem Mandanten eine Sprache zugeordnet werden kann, so kann die Internationalisierung auch direkt durch angepasste Texte in den Templates selber erfolgen. Sowohl bei mehrfach auftretenden Textschnipseln als auch bei Textschnipseln in Templates, die eigentlich nicht überschrieben werden müssen, macht es aber dennoch Sinn, diese in Messages auszulagern.

In Abschnitt 3.3, „Resourcen" wird beschrieben, wie diese Nachrichten konkret bei Buch.de aufgelöst werden.

2.25. Bilder aus Text

Das bei älteren Browsern eingebettete Schriftarten nicht verfügbar sind oder es lizenzrechtliche Schwierigkeiten gäbe, gibt es die Möglichkeit Textschnipsel als Bild zu generieren.

Pagstract selber stellt Konstrukte bereit, die die entsprechenden URLs zusammenbauen. Die eigentliche Arbeit übernimmt dann das dahinterliegende Servlet.

Dabei wird zunächst unterschieden, ob der Text aus einer Property kommt (pma:text-img) oder konstant ist (pma:constant-text-img). Wichtig ist außerdem, dass zumindest der Style gesetzt ist. Dieser muss dann für das Servlet auch konfiguriert sein.

Mit pma:prefix und pma:postfix kann man vorne und hinten an den String etwas anhängen.

Der pma:is-longlife gibt dem Servlet einen Hinweis darauf, ob sich das Cachen lohnt. Bei pma:constant-text-img ist dort der Standardwert true.

Mit pma:maxwidth wird dem Servlet der Hinweis gegeben, das Bild nicht breiter als die angegebene Breite in Pixeln zu rendern. 

<pma:text-img pma:name=".name"
              pma:id="asd"
              pma:style="superStyle"
              pma:prefix="Deine Name ist "
              pma:postfix="!"
              pma:is-longlife="false"
              pma:maxwidth="200"/>
<pma:constant-text-img
              pma:text="Buecher"
              pma:style="superStyle"
              pma:maxwidth="300"/>

<img alt="Buecher" src="http://sample-baseurl.de/imageGenerator?s=superStyle&t=Buecher&c=fs7sX4rawOuCQ&il=true&mw=300&it=NORMAL" >

3. Mandantenspezifische Dateien

Die Ordnerstruktur ist so ausgelegt, dass es einen Basisresourcenordner und mandantenspezifische ResourcenOrdner gibt. Templates, Resourcen und Messages werden zunächst in dem spezielstem Resourcenordner gesucht. Wenn etwas passendes gefunden wurde, endet die Suche. Sonst wird im Zweifelsfall die Resource/das Template/die Message aus dem Basisresourcenordner genommen.

Jeder dieser Resourcenordner inklusive dem Basisresourcenordner hat typischerweise folgende Struktur (wobei einzelne Bereiche, die nicht überschrieben werden, fehlen können):

  • css, img, js und weitere Ordner mit statischen Resourcen, die über resource:// referenziert werden können (siehe Abschnitt 2.22, „Resourcenverweise" )

  • templates: hier wird nach Templates und Tiles gesucht

  • gesucht messages.properties: über diese Datei werden Nachrichten die mit msg:// referenziert werden, aufgelöst (siehe Abschnitt 2.28, „Internationalisierung" )

3.1. Aufloesen von mandantenspezifischen Resourcen

Für das Auflösen von mandantenspezifischen Resourcen/Templates/Messages gibt es zwei Möglichkeiten.

Mögliches Beispiel für Mandant 2:

  1. shop/mandant/2 existiert nicht, daher wird weiter nach einem Verweisordner gesucht.

  2. shop/mandant/2.thalia wird gefunden. Hier wird zunächst nach Templates, Tiles oder Resourcen geguckt.

  3. shop/mandant/thalia muss existieren. Hier wird als zweites nach Templates, Tiles oder Resourcen gesucht.

  4. shop/base Als letztes wird, wie immer, hier nachgeschaut.

3.2. Templates

Die Template-Dateien und Tiles-Dateien (mit pma:tile eingebunden, siehe Abschnitt 2.26, „Verschachteln von Pagstract-Dateien" )werden relativ zu den oben erläuterten Ordnern in den jeweiligen templates-Unterordnern gesucht.

3.3. Resourcen

Resourcenverweise (resource://, siehe Abschnitt 2.22, „Resourcenverweise" ) werden fast genauso aufgelöst wie Templates, nur eben nicht in den templates-Unterordnen, sondern direkt unter dem im Resourcenstring angegebenen Ordner wie css, img etc.

3.4. Nachrichten

Nachrichtenverweise (msg://meine.nachricht, siehe Abschnitt 2.28, „Internationalisierung" ) werden über messages.properties-Dateien direkt in den entsprechenden Resourcenordnern mandantenspezifisch konfiguriert. Es kann natürlich auch nur ein Teil der Messages überschrieben werden — eine messages.properties-Datei muss also nicht immer alle Nachrichtenschlüssel enthalten.
